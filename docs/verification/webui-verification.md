## W02: JSON stdout for binary communication

**Decision says:** "WebUI calls `ksu.exec("zeromount status --json")` and parses stdout. No hex encoding needed."

**VERIFIED from `api.ts`:**

`ksu.exec()` works via a callback-based pattern (verified in `api.ts:30-52`):
```typescript
ksu.exec(cmd, '{}', callbackName);
// callback receives (errno, stdout, stderr)
```

The `'{}' `second argument is an options JSON string. The callback is registered on `window` as a globally-named function.

**Current binary communication:**
- `api.ts` calls individual commands: `zm list`, `zm add`, `zm del`, `zm clear`, `zm enable`, `zm disable`, `zm ver`, `zm blk`, `zm unb`
- NOT calling `zeromount status --json` yet -- that's a v2 design
- Status info is assembled from multiple parallel calls: `getSystemInfo()` fires 7 parallel `execCommand()` calls (`zm ver`, `uname -r`, `cat /proc/uptime`, `getprop ro.product.model`, etc.)
- Status cache: `api.ts:194-213` reads `.status_cache.json` (generated by `monitor.sh` daemon) as a fast path

**Payload limits:** `ksu.exec()` sends a shell command string. The KSU docs don't specify explicit size limits, but shell argument limits apply (typically `ARG_MAX` of ~131072 bytes on Android). The `scanKsuModules()` function in `api.ts:665-704` sends a multi-line shell script (~40 lines) as a single `ksu.exec()` call -- this works but is near practical limits for readability.

**Verdict:** W02 is ACCURATE as a design goal. Current implementation uses per-command calls, not `status --json`. The v2 Rust binary's `status --json` subcommand will simplify this.

## Cross-Validated Findings (with webui-analyst-2)

### Settings persistence pattern gap (implementation-critical)

Current persistence model in `store.ts`:
- `zeromount-theme` -- localStorage only
- `zeromount-accent` -- localStorage only
- `zeromount-fixedNav` -- localStorage only
- `zeromount-autoAccent` -- localStorage only
- `autoStartOnBoot` -- **NOT persisted at all** (not even localStorage)
- `animationsEnabled` -- **NOT persisted at all**
- `verboseLogging` -- localStorage via store + server-side via `api.setVerboseLogging()` (flag file)

**Pattern:** Appearance settings use localStorage. Engine settings are mostly dead. Only `verboseLogging` has dual persistence (localStorage + server-side flag file).

**Implication for BRENE feature toggles (W09):** ALL BRENE toggles control `ksu_susfs` commands that must run at boot time, before the WebUI is opened. localStorage is insufficient because:
1. The Rust binary needs to read toggle state at boot
2. localStorage doesn't survive WebView cache clears
3. Toggle state must persist across reboots

**Required:** BRENE toggles must persist to `config.toml` (per R04) via the Rust binary's `config set` subcommand. The WebUI should call `zeromount config set susfs.auto_hide_apks true` etc., not just update localStorage.

## Additional Findings

### Material Web is a phantom dependency
`@material/web ^2.4.1` and `@material/material-color-utilities ^0.4.0` are in `package.json` but zero Material Web Components are imported or used anywhere in the codebase. All UI is custom SolidJS components. These should be removed from dependencies.

### Version hardcoded in constants.ts
`constants.ts:15`: `APP_VERSION = '3.0.0'`. This is the v1 source code claiming version 3.0.0. The Rust binary (v2) will have version from `module.prop` (per CO04). The constant will become dead code.

### SettingsTab imports GITHUB_URL from constants
`SettingsTab.tsx:8`: `import { GITHUB_URL } from '../lib/constants'` -- points to `https://github.com/backslashxx/zeromount`. Used for the repository link button.

### ksu.exec() callback pattern is fragile
The `api.ts:31-52` callback registration uses `window[callbackName]` global registration. If two calls race with the same callback name (unlikely due to counter), or if the WebView recycles the page, stale callbacks could leak. The v2 approach using `zeromount status --json` stdout parsing will reduce the number of concurrent callbacks.

### Store has dual module systems
`store.modules()` (MountedModule[]) and `store.ksuModules()` (KsuModule[]) are separate signals. The `modules` signal uses `api.getModules()` which reads from a file at `MODULE_PATHS`. The `ksuModules` signal uses `api.scanKsuModules()` which scans `/data/adb/modules/*/` directly. Only `ksuModules` is used in the UI. `modules` is a remnant.
